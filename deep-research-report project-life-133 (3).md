# סיכום מנהלים  
מאגר `project-life-133` הוא **מימוש ייחוס** של מערכת אכיפת תיעוד Policy-as-Code בסביבת פיתוח עם סוכני AI אוטונומיים (כגון Claude Code)【3†L10-L13】【41†L8-L12】. המערכת נועדה למנוע תופעת "תזוזת תיעוד" (Documentation Drift) – מצב שבו הקוד משתנה והתיעוד (קבצי CLAUDE.md, ADRים, JOURNEY.md וכדומה) אינו מתעדכן, מה שעלול לגרום לטעויות ולחוסר עקביות באוטומציה. הפתרון הוא ממשק מדיניות-כקוד (PaC) המחמיר: כל Pull Request עובר בדיקות ב-CI באמצעות **Open Policy Agent (OPA)** (קבצי Rego) ובתסריטים ייעודיים. קיימות ארבע "שכבות" אכיפה: עדכון `CLAUDE.md` בעת שינויי קוד מתחת ל-`src/agent/`; יצירת ADR חדש בעת שינויי תשתית/תלויות; עדכון `JOURNEY.md` לכל שינוי בקוד; ובדיקות עתידיות להתאמת ארטיפקטים חיצוניים (כגון סכמות API)【3†L53-L60】【36†L34-L42】.  
לפרויקט מבנה מסודר הכולל קבצים מרכזיים: `README.md` ו-`CLAUDE.md` המסבירים מטרות וארכיטקטורה, `AGENTS.md` כמסמך תואם לכלי סוכנים (מצביע ל-CLAUDE.md), `JOURNEY.md` כתיעוד ליניארי של סשנים, תיקיית `docs/adr/` להחלטות ארכיטקטוניות (ADRs)【1†L23-L30】【40†L25-L33】, תיקיית `policy/` לקבצי Rego enforcement, סקריפטים בפייתון ו- shell, ועוד רכיבי תשתית (.claude/, Terraform). הקוד העיקרי הוא ב-`src/agent/index.ts` – "מנתב כישורים" (Skills Router) ב-TypeScript חסר-תלויות, המאתר “כישורים” (קבצי SKILL.md) ומבצע התאמה סמנטית (Jaccard) לפקודות משתמש【4†L112-L119】【13†L21-L30】.

**תלויות ורישיון:** הפרויקט אינו מוסיף ספריות צד שלישי לשעת ריצה (למעט תיקייה נטיעת TypeScript לצורך פיתוח). קובץ `package-lock.json` מראה רק תלותי פיתוח (`typescript`, `@types/node`)【11†L9-L12】. בפועל הקוד פועל ללא תקיית `node_modules` ב-PR (ללא ספריות בזמן ריצה)【13†L9-L12】. סקריפטים בפייתון משתמשים רק בספריית הסטנדרט. לא נראה קובץ רישיון (`LICENSE`) – כנראה קוד פתוח פתור (לא צויין במפורש).

**התקנה, בנייה והרצה:** לאחר שיבוט הריפו ניתן להשתמש **כתבנית** לפרויקט חדש – ישנם שני אופציות (סקריפט bash אוטומטי או "Use this template" בגיטהאב)【1†L34-L43】【1†L49-L57】. הגדרה ראשונית: להפעיל ב-Settings > Branch Protection את בדיקת סטטוס `doc-policy-check` כדרישת שימור ענף【1†L64-L70】. לדוגמה, ניתן להתקין pre-commit מקומי לסיבוב יומן (לבדוק אורך):

```bash
git config core.hooksPath .githooks
python3 scripts/rotate_journey.py --check
```
(זה ימנע commit אם `JOURNEY.md` הגיע לסף המקסימלי של כ-380 שורות)【1†L76-L82】.  
המנוע הראשי כתוב ב-TypeScript: להריץ אותו יש צורך ב-Node.js 14+. דוגמה: 

```bash
ts-node src/agent/index.ts "create a new API endpoint"
```
זה יריץ את הפונקציות `discoverSkills`, `routeIntent`, `activateSkill` של מחלקת המנתב【4†L132-L135】. תהליך ה-CI כולל את הסקריפטים (Python) לעדכון אינדקס ADR (`build_index.py`), ולבדיקת frontmatter ADR, ולבדיקות Rego (`check_policies.py <base_sha> <head_sha>`)【2†L23-L29】【3†L37-L44】. הרצתו דרך GitHub Actions מתוארת ב-`.github/workflows/documentation-enforcement.yml` כמבוי חסום (Blocking gate): מיזוג חסום אם כללים עוברים (נכשלת)【3†L99-L106】.

**אבטחה ופרטיות:** הפרויקט משתמש בשיטות מודרניות להגנה על סודות: לכל סקריפט CI משמעות WIF (Workload Identity Federation) מ-GCP במקום אסימון קבוע【45†L14-L23】. קיים חשבון שירות (`github-actions-sa`) עם הרשאות צרכים מותאמים (רק WorkloadIdentityUser וסודיון ספציפי) בלבד【42†L29-L35】【45†L14-L23】. ניהול אסימונים נעשה דרך GCP Secret Manager (לדוגמה, מפתח פרטי של GitHub App מאוחסן כסוד)【45†L41-L46】【45†L67-L75】. אין שימוש ב-PAT ארוך חיים במרבית המקרים; קיימת תלות לטוקן כתיבה (`PUSH_TARGET_TOKEN`), אך הוא נושא סיכון ידוע ויש לתעדו ב-ADR【45†L58-L65】. קוד הסוכן מוגבל לכלים המותרים בלבד (`allowedTools` ממפרט YAML) וסריקת מילות מפתח בלבד משמשת למניעת חישובים חיצוניים כבדים【4†L122-L129】【13†L17-L25】. 

**תהליכי תרומה וניהול:** אין קובץ CONTRIBUTING רשמי, אולם נהלי הפרויקט מבוססים על ADR. כל שינוי מבנה או תלות חדש דורש יצירת ADR חדש ב-`docs/adr/` לפי פורמט MADR (עם frontmatter) והפעלת הסקריפט `validate_adr_frontmatter.py`【3†L95-L103】. בנוסף, יש להריץ `build_index.py --check` לוודא שהאינדקס ב-`docs/adr/README.md` מעודכן【3†L97-L104】. כללי התיעוד המפורטים ב-CLAUDE.md מחדדים את המחויבות: למשל, שינוי בקוד ב-`src/agent/` צריך ללוות עדכון ב-`CLAUDE.md`, שינויי תשתית/תלויות מחייבים ADR חדש, וכל שינוי בקוד מחייב עדכון ב-`JOURNEY.md`【3†L64-L71】. ה-Workflows מותקנים כחובה ב-Branch Protection כך ש-Merge יתאפשר רק לאחר מעבר הבדיקות【3†L99-L106】. הניהול מתועד היטב; בשרשור שיחות (PRs/Issues) מרובים (ניתן לראות PR #24, תחקיר GitHub App למעלה), ויש ענף ראשי פעיל (`main`). המגמה הכללית היא פיתוח תוסס ומורכב אך מסודר.

**בגרות, יציבות ובדיקות:** הפרויקט פעיל עם מאות קומיטים (למשל כ-351 קומיטים מאז תחילת 2025【49†L39-L42】) ומערכת CI מוכחת. כללי האכיפה נאכפים בפועל (נבדק שחסימת מיזוג אכן פעלה פעם ב-24.4.2026 עקב חוסר ADR**【41†L28-L30】). יחד עם זאת, אין במאגר מיצוי אוטומטי של בדיקות תוכנה (unit tests) או CI לבדיקת תפקודיות קוד הסוכן עצמו. תיעוד תזרימי נמצא בפורמט ADR עקבי【40†L25-L33】, ואין "באגים ידועים" מתועדים בקוד – כל שינוי נעשה באמצעות PR/ADR מבוקר. הפרויקט חדש יחסית (רוב העבודה הייתה ברבעון הראשון של 2026【40†L27-L33】) אך כבר בשלב מתקדם של הפקה (מנוע יציב, תשתיות מותאמות).  

**צעדים מומלצים להמשך:** כפי שמתועד ב-CLAUDE.md וב-ADRs, יש כמה פעולות בראש סדר העדיפויות: להטמיע את נסיון ה-Conftest (מבחן אוטומציה פוליסי) בכללים הפעילים (פיילוט ב-`agent_doc_sync.rego`)【4†L149-L158】【41†L28-L30】, להסיר ענפים מיושנים (למשל הסעיף למחיקת `claude/fix-gcp-secret-manager-1gJf2`)【4†L153-L158】, לפרמט ולהרחיב את כלי תשתית (הזזת branch-protection ל-Terraform, הסרת PAT לטובת GitHub App באופן מלא)【4†L153-L158】【45†L58-L65】, ולנקות פרמטרים קשיחים (כמו `ORG_ID`) מהפקודות לפרויקטים אחרים. בנוסף, מומלץ להמשיך להגדיר ADR לכל נושא אדריכלי (כפי שנעשה ב-ADRs 0001–0013【40†L25-L33】), ולתעד את כללי הקוד השוליים (כגון איכות הסקריפטים ב-`policy/`).

**אלטרנטיבות ופרויקטים דומים:** לא קיים פרויקט סטנדרטי מוכר בישראל/בעולם שעושה בדיוק את אותה המשימה. כן קיימים כלי שיתופיים לתיעוד AI (כגון .cursor בקוד העיצוב של Cursor או `.continue/rules` של Continue, כפי שמתואר במקורות מקצועיים【59†L119-L125】). כמו כן, חברות כמו Dosu מפתחות פתרונות לסריקת docs מבוססת AI【28†L75-L83】. אולם `project-life-133` הוא ייחודי בכך שהוא משלב מדיניות-כקוד (OPA) עם תיעוד אינטגרלי (CLAUDE.md, ADR) ו-branch protection כחובה. מיזמים מקבילים בסדרת "Project Life" (למשל 130, 134) עוסקים בפרויקטים מורכבים דומים ומוכיחים רכיבים משותפים (תשתית GCP, GitHub Apps), אך אין תחרות ישירה. כפי שמפורטים ב-ADRs ובמסמכים הרשמיים, הפרויקט עצמו משמש כתבנית (template) לפרויקטים עתידיים המחייבים סוכני AI【1†L34-L43】.
